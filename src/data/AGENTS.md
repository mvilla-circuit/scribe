# AGENTS.md — src/data

The data layer: **React Query** hooks for books, folders, and documents, talking
to Supabase. See the root [`AGENTS.md`](../../AGENTS.md) for repo-wide conventions.

## Area-specific rules

- **Boundaries**: `data` may import from `fonts` and `lib` only. It must not
  reach up into `components`, `editor`, `store`, or `theme`.
- **JSDoc**: exported hooks and types require a description block (no `{type}`
  tags), per the root conventions.
- **Generated types**: query/mutation payloads build on `src/lib/database.types.ts`,
  which is generated by the Supabase CLI — never hand-edit it.

## Caching & mutations

Lessons baked into the current hooks — keep them intact:

- **Keep bulky bodies out of list queries**: structural/list queries
  (`useDocuments`) select only the columns the tree, outline, and breadcrumb
  need. A document's `content` lives in its own `["document-content", id]` query
  so a body autosave never re-downloads every page. Don't fold `content` back
  into the list query.
- **Scope invalidations to the affected key**: invalidate the narrowest key that
  actually changed; never gate an invalidation on a global `qc.isMutating()`
  count, since an unrelated in-flight mutation will suppress it. Reserve the
  page-index invalidation for rename/move/delete/icon — not content saves.
- **Upsert for first-write paths**: a row that may not exist yet (e.g. a profile
  on the first global font pick) must use `upsert`, not `update` — `update`
  silently affects zero rows when none exists. Build the optimistic row when none
  is cached instead of returning the previous (null) value unchanged.
- **Never stamp an empty `user_id`**: optimistic inserts use
  `requireUserId(session)` (or bail) so RLS-bound ids are never `""` during a
  fast double-action.

## Testing

- **Test-first**: follow the repo-wide TDD workflow in the root
  [`AGENTS.md`](../../AGENTS.md) — write the failing test for a hook's behavior
  and watch it fail before implementing it.
- Data-layer tests intercept the **HTTP boundary with MSW** rather than mocking
  the Supabase client. Handlers live in `src/test/msw`; global wiring is in
  `src/test/setup.ts`.
- Tests run against dummy Supabase env values defined in `vite.config.ts`
  (`VITE_SUPABASE_URL=http://supabase.test`, `VITE_SUPABASE_ANON_KEY=test-anon-key`).
  MSW handlers match against that URL, so keep them in sync.
- Use the `render-with-query` helper (`src/test/render-with-query.tsx`) when a
  test needs a React Query provider.
